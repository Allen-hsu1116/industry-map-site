# Spec: Industries / Company Knowledge Generation Pipeline

## Objective

Build a formal, source-traceable pipeline for `industries.json`, `companies.json`, and `company-knowledge/{code}.json` so the site can provide aistockmap-style company pages without letting daily LLM prose become the source of truth.

The pipeline must answer four questions for every company-topic relation:

1. **What does the company actually sell or operate?**
2. **What role does it play in this topic / supply chain?**
3. **How direct is the relationship?**
4. **Which source proves each important claim?**

## Non-Goals

- Do not regenerate the whole topic map every day.
- Do not rewrite SWOT from short-term price movement.
- Do not use an LLM answer as canonical evidence.
- Do not show pipeline/debug labels such as "每日題材定位校正" in user-facing UI.

## Source Priority

1. **Structured market / financial data**
   - FinMind datasets: price, institutional, margin, valuation, revenue, financial statements, balance sheet, cash flow, market value.
   - TWSE / TPEx / MOPS open data.
2. **Company disclosures**
   - Annual reports, quarterly reports, investor presentations, earnings call materials, product pages, sustainability reports.
3. **Regulatory / event data**
   - MOPS major news, dividend announcements, capital expenditure, plant expansion, board resolutions.
4. **Trusted industry/news sources**
   - DIGITIMES, TrendForce, MoneyDJ, 鉅亨, 工商時報, 經濟日報, customer/supplier disclosures.
5. **Reference topic maps**
   - aistockmap and similar maps are candidate/benchmark sources, not final truth.
6. **LLM processing**
   - Allowed: extraction, classification, summarization, contradiction checks, candidate generation.
   - Forbidden: unsourced canonical claims.

## Canonical Data Model

### Topic map: `public/data/industries.json`

`industries.json` remains the canonical topic graph used by the UI.

Each topic should contain:

- `slug`: stable topic id.
- `name`: category + topic display name.
- `description`: what the topic means.
- `groups[]`: supply-chain layers.
- `groups[].companies[]`: company-topic roles.

Each company-topic role should contain:

- `code`, `name`
- `role`: concise supply-chain role.
- `relevance`: `極高 | 高 | 中 | 低` or numeric score only after normalization.
- `analysis`: why this company belongs in this topic.
- `products[]`: product/technology entries.
- `customers[]`: key customers/demand drivers if source-backed.
- `tech_focus[]`: topic-specific technology focus.
- `swot`: topic-specific SWOT, not generic company SWOT.

### Company index: `public/data/companies.json`

`companies.json` should be generated from `industries.json`, not hand-maintained independently.

For each company:

- `code`, `name`
- `topics[]`: topic slugs discovered from the topic graph.
- `topic_count`: `topics.length`.

This prevents the search index from drifting away from the topic map.

### Company knowledge: `public/data/company-knowledge/{code}.json`

Generated by `scripts/generate-company-knowledge.ts` from financial snapshots and topic analysis.

It is a company-level rollup containing:

- products
- customers
- topicRoles
- SWOT with freshness metadata
- data source availability
- FinMind / MOPS coverage signals

## Best Product Description Strategy

The current frontend helper can explain common product names like CoWoS or N2, but that is only a display fallback. The best long-term approach is to make product descriptions first-class sourced knowledge.

Current slice:

- `public/data/product-knowledge/2330.json` is the first evidence-backed product knowledge sample.
- `src/lib/productKnowledge.ts` defines the reusable schema and lookup/normalization helper.
- The company detail UI fetches `/data/product-knowledge/{code}.json`, renders sourced product narratives first, and only falls back to `describeProduct()` when no knowledge entry matches.
- `npm run knowledge:validate` validates product knowledge shape and checks that each evidence `sourceId` exists in `knowledge-sources.json`.

Each product should be an object instead of a string:

```json
{
  "name": "CoWoS-S",
  "aliases": ["CoWoS", "CoWoS-S + HBM3E"],
  "category": "advanced_packaging",
  "plainLanguage": "A 2.5D advanced packaging route using a silicon interposer to connect AI accelerators and HBM.",
  "whyItMatters": "It reduces the data-movement bottleneck between compute and memory, making it central to high-end AI GPU performance and supply.",
  "topicFit": {
    "cowos-advanced-packaging": "Direct product/technology platform for AI advanced packaging.",
    "hbm": "Key integration path between AI accelerators and HBM."
  },
  "businessImpact": "CoWoS capacity and yield affect AI/HPC customer shipment timing and support TSMC's advanced packaging pricing power.",
  "evidence": [
    {
      "sourceId": "tsmc-cowos-platform",
      "url": "https://www.tsmc.com/english/dedicatedFoundry/technology/cowos",
      "publisher": "TSMC",
      "title": "CoWoS advanced packaging technology platform",
      "retrievedAt": "2026-05-27",
      "claim": "TSMC identifies CoWoS as an advanced packaging platform for HPC/AI applications."
    }
  ],
  "lastVerified": "2026-05-27",
  "confidence": "high"
}
```

Why this is better than a frontend helper:

- The explanation is reusable across UI, daily analysis, reports, and future APIs.
- Every claim can cite evidence.
- Product wording can differ by topic: CoWoS in `cowos-advanced-packaging` is direct; CoWoS in an equipment topic may be a demand driver.
- Updates can be event-driven: product page change, annual report, law conference, customer ramp, capacity expansion.
- The frontend becomes a renderer, not a hidden knowledge generator.

## Pipeline Stages

### Stage 1: Candidate discovery

Inputs:

- Existing `industries.json`.
- Reference maps such as aistockmap.
- News and industry sources.
- Company disclosures.

Output:

- Candidate topic/company/product facts with URLs and snippets.
- No canonical JSON mutation yet.

### Stage 2: Evidence extraction

Use deterministic parsers where possible and LLM extraction only as a structured transformer.

Output schema per claim:

- `claimType`: product | customer | role | capacity | financial | risk | opportunity.
- `claim`: concise statement.
- `evidenceUrl`, `publisher`, `publishedAt`, `snippet`.
- `companyCode`, `topicSlug` if applicable.

### Stage 3: Role classification

Classify each company-topic relation:

- `direct_platform`: company sells/operates the product/platform that defines the topic.
- `direct_supplier`: company supplies a key component/material/equipment.
- `direct_customer`: company is a major demand driver.
- `indirect_beneficiary`: topic benefits the company through customers/market expansion, but not a direct role.
- `peripheral`: weak or speculative relation.

`relevance` should be derived from evidence-backed classification:

- `極高 / high`: direct platform, monopoly/oligopoly, or indispensable bottleneck.
- `高`: direct supplier/customer with repeated evidence.
- `中`: real but not core, or revenue exposure unclear.
- `低`: peripheral, optional, or mostly market narrative.

### Stage 4: Canonicalization

Human/agent review promotes extracted claims into:

- `industries.json`
- future product knowledge files
- financial snapshots' `industry_analysis`

Rules:

- Every new high-impact role/SWOT/product must have at least one evidence record.
- Conflicting claims are marked `needs_review`, not silently overwritten.
- Old claims remain until explicitly superseded.

### Stage 5: Validation

Run:

```bash
npm run knowledge:validate
npm run knowledge:build
npm run analysis:daily
npm run build
```

Validation should check:

- topic slugs are unique.
- companies referenced in `companies.json` exist in `industries.json` and vice versa.
- `topic_count` equals discovered topics.
- roles have code/name/role/relevance.
- product descriptions are not just bare nouns.
- warnings are written to `public/data/knowledge-quality-report.json`.

### Stage 6: Daily / event refresh

Daily refresh updates market data and generated analysis only.

Event-driven refresh updates company/topic knowledge when:

- MOPS major news affects products, capacity, dividends, capex, litigation, M&A, or operations.
- Financial reports reveal segment/portfolio changes.
- Monthly revenue or margins materially diverge from topic thesis.
- Trusted industry news confirms customer ramps, product cancellations, or technology changes.

## Success Criteria

- `industries.json` and `companies.json` no longer drift silently.
- Product explanations are treated as canonical sourced knowledge, not frontend-only hacks.
- Daily analysis consumes stable knowledge and market data, but does not rewrite the knowledge base.
- UI can show source/freshness metadata without exposing pipeline implementation language.
- Validation produces a machine-readable quality report for follow-up cleanup.

## Implementation Slices

1. **Foundation validation**
   - Add `scripts/validate-knowledge-pipeline.ts`.
   - Add `npm run knowledge:validate`.
   - Write quality report JSON.
2. **Generated company index**
   - Add script to regenerate `companies.json` from `industries.json`.
   - Replace manual edits with generated output.
3. **Source/evidence registry**
   - Add `public/data/knowledge-sources.json` and per-claim source ids.
   - Start with 2330 / CoWoS as the first high-quality case.
4. **Product object migration**
   - Introduce product object schema while preserving string backward compatibility.
   - Move frontend `describeProduct()` fallback into generated knowledge.
5. **Event scanner**
   - Add major-news/news scanner that emits `needs_review` deltas.
6. **UI source display**
   - Show source/freshness chips per product/SWOT/role when evidence exists.

## Boundaries

- **Always:** prefer official and structured sources.
- **Always:** keep LLM output labeled as extracted/derived, never source truth.
- **Always:** preserve deterministic builds from static JSON.
- **Ask first:** paid APIs, DB services, external queues, or non-static backend infrastructure.
- **Never:** fabricate citations, silently overwrite SWOT, or make product claims without source provenance.
