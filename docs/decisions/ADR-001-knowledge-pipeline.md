# ADR-001: Treat Industry and Company Knowledge as Versioned Evidence, Not Daily LLM Output

## Status

Accepted

## Date

2026-05-27

## Context

The site is evolving from a static topic map into a company analysis product. Users expect company pages to explain products, topic roles, SWOT, and why a company matters inside each theme, similar to aistockmap.

The previous implementation had two risks:

- Topic/product/SWOT knowledge could be mixed into daily analysis prose, making it unclear what is stable knowledge vs. a same-day market read.
- Product explanations were partly generated by frontend fallback helpers, which improves display quality but is not a real source-traceable knowledge base.

We need a pipeline that can ingest evidence, normalize it, validate it, and only then expose it to daily analysis and UI.

## Decision

Use a three-layer knowledge architecture:

1. **Canonical topic graph**: `public/data/industries.json`
   - Defines topics, groups, company roles, relevance, products, customers, tech focus, and topic SWOT.
2. **Generated company index/rollup**:
   - `public/data/companies.json` should be generated from the topic graph.
   - `public/data/company-knowledge/{code}.json` is generated from financial snapshots and topic analysis.
3. **Daily analysis**:
   - Consumes canonical knowledge and daily market data.
   - Does not rewrite canonical roles, product descriptions, relevance, or SWOT.

LLMs may be used only as extraction/classification/summarization workers. They cannot be the cited source for canonical facts.

## Alternatives Considered

### Let daily analysis regenerate product/topic/SWOT each day

- Pros: simple to prototype, always fresh-looking.
- Cons: nondeterministic, hard to trace, likely to drift into hallucinated topic relations.
- Rejected because user-facing investment analysis needs source provenance and stable semantics.

### Keep all product explanations in frontend helpers

- Pros: fast UI improvement, no data migration required.
- Cons: knowledge is hidden in UI code, cannot cite sources, cannot feed reports/API consistently.
- Rejected as canonical approach. Acceptable only as a temporary fallback.

### Move immediately to a database-backed knowledge system

- Pros: better versioning and review workflow.
- Cons: adds infra and operational scope before the static JSON pipeline is stable.
- Deferred. Static JSON plus validation is the right next step.

## Consequences

- Validation becomes a required step before regenerating daily analysis.
- Future product descriptions should become evidence-backed objects, not plain strings.
- `companies.json` drift must be detected and eventually removed by generating it from `industries.json`.
- UI should display stable knowledge cleanly and hide internal pipeline labels.
- The first implementation slice is intentionally static-file based and rollback-friendly.
